# Project: letterkit

Build the foundational monorepo for **letterkit** вЂ” an open-source, copy-paste component library for transactional emails. Think "shadcn/ui for React Email." Developers run a CLI to copy block source files directly into their project, not install from npm.

## Core Philosophy
- **Copy, don't install.** Blocks are owned by the user, fully editable.
- **React Email + Tailwind** as the foundation.
- **Registry-driven:** a JSON manifest indexes all blocks; CLI reads from a hosted registry URL.
- **Theme-first:** all blocks consume design tokens (colors, spacing, radius, fonts) from a `theme.ts` file.
- **Email-client tested:** every block must work on Gmail, Apple Mail, Outlook (incl. Outlook Windows), iOS Mail, Yahoo.

## Tech Stack (do not deviate)
- pnpm workspaces + Turborepo
- TypeScript strict mode everywhere
- Biome for lint + format (no ESLint, no Prettier)
- Changesets for versioning
- tsup for package builds
- Next.js 15 (App Router) + Tailwind v4 for `apps/www`
- React Email (`@react-email/components`, `@react-email/render`) for blocks
- Commander.js for the CLI
- Vitest for tests
- Node 20+

## Implementation Order

### 1. Workspace bootstrap
- Root `package.json` with workspace scripts: `dev`, `build`, `lint`, `typecheck`, `test`, `changeset`.
- `pnpm-workspace.yaml` includes `apps/*` and `packages/*`.
- `turbo.json` with pipelines for build/dev/lint/typecheck/test (with proper `dependsOn`).
- `tsconfig.base.json` with strict mode, path aliases.
- `biome.json` configured for TS/React, 2-space indent, single quotes, semicolons, line width 100.
- `.gitignore` covers node_modules, .next, .turbo, dist, .DS_Store, .env*.

### 2. @letterkit/theme
- Export typed design tokens: `colors`, `spacing`, `radius`, `fontFamily`, `fontSize`.
- Two default themes: `light` and `dark`.
- Tokens are plain TS objects, no runtime dependency.

### 3. @letterkit/registry
- Zod schemas: `BlockSchema`, `BlockCategory`, `BlockManifest`.
- Each block has: `name`, `category`, `description`, `dependencies` (npm), `registryDependencies` (other blocks), `files` (path + content), `previewProps`.
- Build script reads `blocks/**` directory, validates each block, outputs `registry.json` at package root.
- Export `getBlock(name)`, `listBlocks()`, `listCategories()` helpers.

### 4. @letterkit/cli
- Binary name: `letterkit`.
- Commands:
  - `letterkit init` вЂ” creates `letterkit.json` config (paths, theme location, tsx/jsx).
  - `letterkit add <block-name>` вЂ” fetches from registry, writes files to user's project, installs npm deps.
  - `letterkit list` вЂ” lists available blocks grouped by category.
  - `letterkit diff <block-name>` вЂ” shows what changed if upstream block updated (stretch goal, scaffold but mark TODO).
- Registry URL is configurable via env `LETTERKIT_REGISTRY_URL`, defaults to a placeholder `https://letterkit.dev/registry`.
- Use `prompts` for interactive flows, `picocolors` for output, `ora` for spinners. NO chalk (deprecated maintainer drama).

### 5. The 6 initial blocks
For each block in the structure above, create:
- `index.tsx` вЂ” the React Email component, fully typed props, JSDoc with usage example.
- `preview.tsx` вЂ” default props for preview rendering.
- `meta.json` вЂ” manifest entry (category, description, dependencies).

Quality bar for blocks:
- Mobile-first (single column, 600px max width).
- Use only Tailwind classes that React Email supports (`tailwind-config` aware).
- Pass props through, never hardcode brand strings.
- Dark mode handled via `@media (prefers-color-scheme: dark)` CSS in `<Head>`.
- Include preheader (`<Preview>`).
- Include accessible alt text on all images.
- Plain-text fallback considered (mention in JSDoc).

### 6. apps/www
- Next.js 15 App Router, Tailwind v4, shadcn/ui for the site UI.
- Pages:
  - `/` вЂ” landing: hero with CLI command, "Why letterkit" 3-column, block gallery preview, CTA.
  - `/docs` вЂ” getting started, CLI reference, theming guide, contributing.
  - `/blocks` вЂ” gallery view of all blocks (read from registry).
  - `/blocks/[category]/[name]` вЂ” single block: live preview (iframe rendering React Email output), code tab, props tab, copy CLI command.
- Render preview by importing block and using `@react-email/render` on the server.
- Dark mode toggle.
- No auth, no DB. Static + RSC.

### 7. CI
`.github/workflows/ci.yml`:
- Triggers: push to main, all PRs.
- Jobs: install (pnpm with cache), lint (`biome check`), typecheck (`turbo typecheck`), test (`turbo test`), build (`turbo build`).
- Node 20, pnpm 9.

## Non-negotiables
- No `any` types. Use `unknown` and narrow.
- No default exports except for React components where React Email expects them.
- All packages have proper `exports` field in `package.json` (ESM only).
- All packages have `"type": "module"`.
- Every public function/component has JSDoc.
- Commit messages follow Conventional Commits.

## What NOT to do (yet)
- No tests beyond a smoke test per package вЂ” we'll add coverage later.
- No paid "Pro" blocks logic вЂ” that comes later via a separate private registry.
- No email-sending integration вЂ” letterkit only produces templates.
- No visual editor вЂ” out of scope for v0.
- No i18n system вЂ” blocks are English-only for v0.
